# CS-255-Sys-Analysis-and-Design

The client for this project was DriverPass, which is a startup driving school owned by Liam, and Ian handling the IT side. They requested a system that could bring everything together into one place instead of doing things manually. This included registering students, scheduling lessons across their 10 cars, and managing their three different training packages. The main goal was to build a cloud based system that was easy to use and accessible from anywhere.

What I Did Well:

I think one of the things I did really well was taking the real world rules of the business and turning them into different, clear system designs. Tthe 10 car limit could have caused scheduling issues, but  tried to make sure that it was handled properly in my diagrams. My sequence diagram shows how the system checks availability before confirming a lesson, which helps prevent double-booking.

Areas for Revision:

If I could go back and improve something, I would spend more time on my class diagram. At first, I focused more on getting the structure down, but I could have added more detail. Especially for the practice test part of the system. If I revised it I would include more specific attributes and data types to make it clearer for developers who would actually build the system.

Interpreting User Needs:

I focused a lot on what the users were struggling with, especially things like scheduling and keeping track of lessons. The secretary’s challenges actually stood out, so I made sure the system handled scheduling in a way that avoids conflicts and keeps everything organized. I feel some real world experience came in handy there. It’s important to design around user needs because even if a system works technically, it won’t be useful if it doesn’t actually help the people using it.

My Design Approach:

When I design a system, I start by identifying the main pieces (like users, lessons, and payments) and then figure out how they all connect. I also look at what actions need to happen and how the system should respond. One thing I found helpful was using activity diagrams early on to map out the logic before getting too deep into the design. Moving forward, I’d keep using that approach because it helps catch problems early and makes everything easier to build later.
